# ArcherReview — AI Adaptive Study Calendar (NCLEX) — Upgrade Plan

> Generated from a senior multi-disciplinary audit (product · AI/ML · backend · frontend · architecture). Roadmap to take this project to its expected full form. Tackle order across all 3 repos: **Othor AI → ArcherReview → CryptAI**.

## Vision (full form)
ArcherReview's full form is a trustworthy, production-deployed AI adaptive study platform for NCLEX: a learner registers, takes a content-backed adaptive diagnostic, and receives a personalized plan on an interactive drag-and-drop calendar; each day they open a task, watch/read/quiz in-app, get rationales, and mark completion with confidence; completing tasks and rescheduling visibly triggers the monitor/adaptation/remediation agents, which rebalance the plan and inject targeted review, with every change surfaced in a human-readable adaptation log. A progress dashboard shows a REAL computed readiness score, topic mastery, and trend data derived from actual performance, and a RAG-grounded AI tutor answers clinical questions with sourcing. It runs on enforced server-side auth, durable DB-backed scheduling via real cron, validated inputs, quality-gated CI, and containerized deployment.

## Current state
Substantial, above-prototype build that demos well on the surface but breaks the moment a reviewer engages the core loop, and carries critical security/integrity defects. VERIFIED in the code: (1) /quiz/[id] is referenced from 5 call sites (TaskDetailPopover.tsx:154, LazyTaskDetailPopover.tsx:145, calendar/page.tsx:750, dashboard/page.tsx:634, RemediationPrompt.tsx:79) but no src/app/quiz route exists -> guaranteed 404 on the product's central action. (2) Dashboard/progress analytics are hardcoded arrays in monitorAgent.ts (performanceTrend=[65,68,72,...] and ~10 mock* arrays, lines 774-857, comment 'until real data is available'). (3) Gemini key read from NEXT_PUBLIC_GOOGLE_GENERATIVE_AI_KEY in generativeAI.ts:5 and agentAI.ts:6 (client-inlineable, billable-key foot-gun). (4) Broken access control: only 4 of ~44 userId-handling API routes verify a token, 40 read userId from query/body, authMiddleware.ts is imported by ZERO routes, and there is NO root middleware.ts. (5) Both auth systems default JWT_SECRET to 'your-secret-key' (auth.ts:5, auth-options.ts:60). (6) Adaptivity not wired to UX: tasks/[id]/complete/route.ts (68 lines) only sets status='COMPLETED'; calendar handleTaskUpdate only PUTs new time -> no monitor/adaptation/remediation invoked. (7) agentScheduler uses module-level in-memory `let scheduleEntries` (line 42), wiped on serverless cold start; agentInitializer is import-triggered as a side effect in layout.tsx:10 (DB work on render path). (8) schedulerAgent uses Math.random() for review-topic (line 377) and task-type (line 446) selection. (9) Diagnostic is hardcoded sampleQuestions in OnboardingContext.tsx:96. ## Runs today?
Partially. node_modules is not installed, so a fresh clone needs `npm install` first. There is no .env.example; required vars are NEXT_PUBLIC_GOOGLE_GENERATIVE_AI_KEY (rename to server-only), MONGODB_URI, JWT_SECRET, OPENROUTER_API_KEY, optional USE_LLM_* flags and ADMIN_KEY. With those set, `npm run dev` (Turbopack) starts and the build 'passes' only because next.config.mjs suppresses all TS/ESLint errors. Landing, onboarding, dashboard, calendar (drag-drop), tutor chat, and progress charts render, but clicking any QUIZ task 404s (/quiz/[id] missing) and the charts show fabricated numbers. DB needs seeding via `npm run seed` (scripts/seed-db.js) per DATABASE_SETUP.md.

## Stack
Next.js 15.3.1 (App Router, output:'standalone'), React 19, TypeScript 5 (strict:false), Tailwind v4, Framer Motion, FullCalendar (daygrid/timegrid/interaction), Chart.js, date-fns. Backend: MongoDB via Mongoose 8 (15 models, 61 API routes), custom JWT auth + unused NextAuth. AI: Google Gemini (@google/generative-ai 0.24.1, default gemini-1.5-pro) with OpenRouter HTTP fallback (default qwen/qwen3-coder:free) and keyword fallback; six-agent layer (monitor/adaptation/remediation/feedback/scheduler/evolution) that is mostly deterministic rules + optional LLM (USE_LLM_* flags default OFF).

## Roadmap

### P0 — Fixes (make it correct & runnable)
- [ ] **(L)** Enforce server-side auth on all data routes via a single guard + root middleware.ts; derive userId from a verified token, never from query/body
  - _why:_ VERIFIED mass IDOR: only 4 of ~44 userId-handling routes check a token, 40 read userId from request, authMiddleware.ts is imported by zero routes, and there is no root middleware.ts. Any anonymous caller can read/modify any user's plans/tasks/performance and hit destructive cleanup routes. This is the single biggest blocker to shipping.
- [ ] **(S)** Move Gemini key to a server-only var (GOOGLE_GENERATIVE_AI_KEY) and rotate it
  - _why:_ generativeAI.ts:5 and agentAI.ts:6 read NEXT_PUBLIC_GOOGLE_GENERATIVE_AI_KEY; the NEXT_PUBLIC_ prefix inlines a billable key into the client bundle the instant any client component imports it. Today it's server-only by accident, not by design.
- [ ] **(S)** Fail-fast on missing/default secrets; remove the 'your-secret-key' fallback
  - _why:_ Both auth.ts:5 and auth-options.ts:60 default JWT_SECRET to the public literal 'your-secret-key' -> tokens are trivially forgeable if deployed without the env set. Boot must throw if JWT_SECRET/MONGODB_URI/Gemini key are unset.
- [ ] **(L)** Build the /quiz/[id] runner and a learner Task View (video/reading) and capture per-question performance
  - _why:_ The product's core action 404s: /quiz/${contentId} is linked from 5 call sites but no route exists. Without the view-content -> quiz -> feedback loop there is no actual studying and any demo dies on the first task click. Capturing real answers also unblocks real analytics.
- [ ] **(M)** Replace hardcoded mock analytics with real aggregations from Performance/Task data
  - _why:_ monitorAgent.ts:774-857 returns ~10 fabricated arrays (performanceTrend=[65,68,72,...]) that feed the dashboard and progress charts. A data-driven adaptivity product showing invented numbers is a trust and due-diligence landmine.
- [ ] **(M)** Wire adaptivity into the UX: on task-complete and drag-reschedule, run monitor->adaptation->remediation and surface what changed
  - _why:_ tasks/[id]/complete/route.ts only sets status='COMPLETED' and calendar handleTaskUpdate only PUTs the new time; neither invokes any agent, so the marquee 'AI rebalances your plan' is invisible. This is the core differentiator and is currently dead.
- [ ] **(M)** Re-enable TS/ESLint build gates and fix resulting errors; clean up next.config.mjs
  - _why:_ next.config.mjs sets ignoreBuildErrors/ignoreDuringBuilds:true and has a duplicated `experimental` key (second {} wipes the first) plus deprecated appDir. Real bugs ship silently with strict:false. Gates must be on before any further feature work to stop regressions.

### P1 — Features (reach the full product)
- [ ] **(L)** Replace Math.random() scheduling and fixed-interval spaced repetition with deterministic, performance-driven logic (SM-2/FSRS per-item state)
  - _why:_ schedulerAgent.ts:377 (review-topic) and :446 (task-type) use Math.random(), and spacedRepetition.ts is a coarse fixed-interval table. The docs claim adaptive difficulty and 'advanced spaced repetition with interleaving' that does not exist; plans are non-reproducible run-to-run.
- [ ] **(M)** Make the diagnostic content-backed and adaptive; feed real weak-area signals into plan generation
  - _why:_ OnboardingContext.tsx:96 ships hardcoded sampleQuestions scored client-side, yet UI copy claims an 'adaptive 15-20 minute' assessment. Plan personalization currently rests on a tiny fixed quiz divorced from the content DB.
- [ ] **(M)** Compute a real readiness/forecast model (weighted by NCLEX category) with top-3 weaknesses
  - _why:_ Readiness is a placeholder; evolutionAgent compares first-half vs second-half averages with +/-5% thresholds (no real model). A credible NCLEX product needs a defensible readiness score and forecast surfaced on the dashboard.
- [ ] **(M)** Durable DB-backed agent scheduling triggered by real cron; fix the dead event-trigger path
  - _why:_ agentScheduler.ts:42 keeps jobs in an in-memory array wiped on cold start and not shared across instances, and triggerEventBasedRun creates entries with undefined nextRun that never execute. 'Autonomous monitoring for all users' effectively never runs in production.
- [ ] **(L)** Add structured LLM output (Gemini responseSchema/JSON mode) + zod validation, and a real NCLEX eval harness; replace fake confidence
  - _why:_ LLM JSON is parsed via regex+JSON.parse with no schema/enum checks; llmTesting.ts is keyword/length heuristics (no factuality), and calculateConfidence() derives 'confidence' from response length then persists it as if calibrated. For clinical content this risks shipping confidently wrong nursing guidance.
- [ ] **(L)** Add RAG grounding for the tutor over a vetted NCLEX corpus, and seed a real end-to-end content slice
  - _why:_ No embeddings/vector/RAG exist; the tutor free-generates clinical claims (e.g., hardcoded fallback drug/exam facts) with no sourcing, and the content library is seed-only with no real video URLs. A grounded, citable tutor + one real playable topic is the biggest credibility upgrade.
- [ ] **(M)** Add input validation (zod) at the API boundary and a shared error responder; add DB indexes + transactions
  - _why:_ No validation library; 51 routes return raw error.message (leaks internals); no indexes on the hot Task.plan/topic/startTime queries; zero transactions make multi-doc writes non-atomic, which is why 8 manual duplicate-cleanup routes exist. Fixing root causes lets those cleanup routes be deleted.

### P2 — Polish & cleanup
- [ ] **(S)** Delete app_backup/, page.optimized*.tsx, loose root scripts, test-*/demo routes, and dead react-beautiful-dnd
  - _why:_ 880K app_backup/ duplicate (incl. insecure auth copy), 7 loose root scripts, 5 test-*/demo app routes, and an unused React-19-incompatible dnd dep inflate audit/attack surface and confuse maintainers. Fastest high-signal cleanup.
- [ ] **(L)** Unify on the archer-teal design system and build reusable primitives; refactor indigo/violet/fuchsia screens
  - _why:_ 47 files use off-brand indigo/violet/fuchsia (incl. the main dashboard and MainNav) vs 25 on the documented archer-teal tokens; even the wordmark color is inconsistent. A single token system is needed for a finished, client-ready look.
- [ ] **(M)** Gate developer/cleanup tools behind admin/debug flag; remove user-facing CleanupButton/TimeCleanupTool; fix 'coming soon' dead ends and MainNav active-tab/logout
  - _why:_ Debug cleanup buttons leak into the learner calendar/dashboard, profile/admin show non-functional 'coming soon' placeholders, MainNav isActive never matches (compares pathname to href-with-query) and has no logout. These make the app feel unfinished.
- [ ] **(L)** Add app-router error.tsx/loading.tsx/not-found.tsx, empty/error states, a data layer (SWR/React Query), and an a11y + mobile pass
  - _why:_ ~135 hand-rolled fetch calls (incl. a duplicated ~90-line dashboard block reading stale state), no route boundaries, only 7/102 components use aria-*, and calendar/dashboard are desktop-first though NCLEX students study on phones. Needed for robustness and a real v1 mobile experience.
- [ ] **(M)** Consolidate config and docs; rename package to archerreview; rewrite README + footer; commit .env.example
  - _why:_ Two ESLint configs, two DB modules, conflicting path aliases, 8 overlapping docs (~4000 lines), package name still 'dynamic-calendar', footer says '2024 ... prototype', and no .env.example means a new dev/evaluator cannot stand it up. One source of truth restores professionalism.
- [ ] **(M)** Add a logger (pino), strip 624 console statements and the MONGODB_URI/key-status logs, and add a happy-path E2E test
  - _why:_ Console noise logs user IDs/responses and db.ts echoes the full MONGODB_URI (credential leak in prod logs). A register->onboard->plan->complete->see-adaptation E2E locks the demo flow and would have caught the /quiz 404.

### Deployment
- [ ] **(M)** Add multi-stage Dockerfile (leveraging output:'standalone') + .dockerignore + docker-compose with MongoDB
  - _why:_ No Dockerfile/compose/.dockerignore exist despite next.config already set to output:'standalone' and serverExternalPackages:['mongoose']. Containerization gives local/prod parity and an easy path to any host.
- [ ] **(S)** Set up GitHub Actions CI (install, tsc --noEmit, next lint, build, tests) gating PRs
  - _why:_ No .github/workflows and no test runner exist; with gates re-enabled, CI is what prevents the suppressed-error regressions from returning.
- [ ] **(M)** Deploy to Vercel + MongoDB Atlas with Vercel Cron hitting an ADMIN_KEY-protected /api/agents/process
  - _why:_ The Next app fits Vercel cleanly; Atlas provides managed Mongo; Vercel Cron (vercel.json crons) is the durable trigger that replaces the in-memory scheduler so adaptation actually runs on a cadence in production.
- [ ] **(S)** Pre-deploy hardening: remove/env-gate test-* and demo routes, require admin auth + NODE_ENV guards on cleanup/* routes
  - _why:_ Six test/demo route groups and 8 unauthenticated destructive cleanup routes must not be reachable in production; gating them is a release blocker for a public deploy.

## Key decisions
- Pick ONE auth system. Recommend the existing custom JWT moved into an httpOnly cookie + a single server-side guard and root middleware.ts, and delete the unused NextAuth stack (auth-options.ts) to remove the dual-system confusion.
- Target Vercel + MongoDB Atlas + Vercel Cron as the primary deploy. Keep the Docker/compose path as the portable fallback (e.g., Render/Fly) only if a long-running worker is later needed for the scheduler.
- Persist agent schedules in MongoDB and trigger processing via external cron rather than reviving the in-memory scheduler; also remove the agentInitializer import side-effect from layout.tsx so no DB work happens on the render path.
- Sequence strictly: P0 security (auth, secrets, key) and the core quiz/Task-View loop + real analytics FIRST, because they are both the top demo-blocker and the top trust/security risk; defer SM-2/FSRS, RAG, and readiness modeling to P1 once real performance data is being captured.
- Centralize model selection in one config module, upgrade the default off the deprecating gemini-1.5-pro to a current 2.x model, and switch the OpenRouter fallback from a coding model (qwen3-coder) to an appropriate general/medical-capable model.
- Adopt zod as the single boundary-validation + LLM-output-validation library (paired with Gemini responseSchema), and a shared error responder that logs server-side but returns a generic message + request id.

## Risks
- Auth refactor is the riskiest change: routing identity through a verified session and adding ownership checks touches ~44 API routes plus every client fetch that passes ?userId=. Sequence it as the first P0 with a thin auth helper + middleware, then migrate routes in batches behind tests, or you will break every screen at once.
- Wiring agents into task-complete/reschedule can regenerate duplicate tasks (the very reason 8 cleanup routes exist). Do not enable agent triggers until DB indexes + transactions/idempotency land, or you will amplify the existing data-corruption bug for real users.
- 'AI' marketing vastly overstates reality: core scheduling is rule-based with literal Math.random(), 'spaced repetition' is a fixed table, and 'predictive modeling' is a half-vs-half average. Either build the real algorithms (P1) or soften the docs/landing copy now to survive technical/clinical due diligence.
- Clinical safety: the tutor can emit confidently wrong nursing/drug/dosage guidance with no grounding or factuality eval, and fake length-based confidence is persisted as if calibrated. Until RAG + a real eval harness ship, add a visible disclaimer and avoid presenting tutor output as authoritative.
- Re-enabling TS strict + ESLint will surface a large backlog (~235 any, strict:false, 14 @ts-ignore). Turn gates on incrementally (typecheck-only first, then strict module-by-module) so the build doesn't stay red for days and stall feature work.
- Replacing mock analytics depends on the quiz/Task-View loop existing first (no real performance data is captured today). Build the daily loop before the analytics, or the 'real' charts will simply be empty.
- Operational: in-memory scheduler/cache/rate-limiter all silently no-op on serverless/multi-instance. Anything relying on them (autonomous monitoring, LLM rate limits, dedupe cache) is effectively absent in a Vercel deploy until moved to Mongo/Redis + real cron.

## Estimated scope
Roughly an 8-12 week effort for one strong full-stack engineer to reach a polished, deployable v1. Week 1-3: P0 security hardening (auth/middleware/secrets/key) + re-enable build gates + 1-day repo cleanup. Week 3-6: core daily loop (/quiz/[id] + Task View), real analytics from captured performance, and visible adaptivity wiring. Week 6-10: P1 intelligence (deterministic scheduling/SM-2, adaptive diagnostic, readiness model, durable cron scheduling, structured LLM output + eval, optional RAG/content slice). Week 10-12: P2 design-system unification, a11y/mobile, docs consolidation, plus Docker/CI/Vercel+Atlas deployment with pre-deploy hardening. P0 + deployment alone (a secure, correct, shippable but not-yet-fully-intelligent product) is achievable in ~4-5 weeks.

## Cross-cutting standards (shared across all 3 repos)
- Auth/authorization as one shared pattern: all three route identity unsafely (Othor IDOR + 2 unauthenticated endpoints; Archer derives userId from query/body on 40 of 44 routes with a guard imported by zero routes and no root middleware; CryptAI has none yet). Standardize on JWT-in-httpOnly-cookie + a single server-side guard + an ownership/resource-scoping dependency, build it once in Othor, and port it.
- Secrets hygiene: hardcoded/default signing keys and committed credentials are universal — Othor commits a SQLite DB with a real email+bcrypt hash and hardcodes SECRET_KEY/POSTGRES_PASSWORD; Archer defaults JWT_SECRET to 'your-secret-key' in two auth systems; CryptAI ships realistic sk-... placeholders and weak compose passwords. Standard: fail-fast on missing/placeholder secrets at boot, rotate everything treated as compromised, purge secrets/PII from git history, and adopt a secret store for deploy.
- AI API keys must be server-only: Othor and Archer both inline billable LLM keys into the client bundle via NEXT_PUBLIC_* (Archer's Gemini key is server-side only by accident). Standardize on server-only env names and rotate.
- Env-driven config / no baked localhost: every repo hardcodes URLs or CWD-relative paths (CryptAI's localhost:8000 in App.tsx, Othor's build-time-baked NEXT_PUBLIC_API_URL=localhost:8001 and Path('data/...'), Archer's path-alias/config sprawl). Standardize a single typed settings module per service plus build-arg/runtime injection of the public API URL.
- CI quality gate is absent everywhere and quality gates are actively suppressed in the two Next apps (Othor next.config ignores TS+ESLint; Archer ignoreBuildErrors/ignoreDuringBuilds true). Standardize one GitHub Actions template: typecheck + lint + tests + docker build, and re-enable the suppressed gates early (incrementally) so hidden errors surface before refactors.
- Honest docs / kill fabricated data: each app shows invented numbers or overclaims (Othor's hardcoded 94.2%/0.95 accuracy + README claims of Celery/refresh-tokens/scale that don't exist; Archer's hardcoded performanceTrend arrays and 'AI' that is Math.random; CryptAI's docs describing a finished platform that is 97% absent). Standard: replace mock data with real fetches or true empty states, and trim copy to an 'implemented vs planned' table.
- Deploy shape is convergent: two Next apps -> Vercel (or container) + managed Mongo/Postgres + object storage + cron for background work; CryptAI is the outlier (stateful, multi-DB, WebSocket, long-running loop -> container platform/VM, never serverless). Reuse one container-platform + managed-data + CI/CD playbook across all three.
- Background/async work + durable scheduling: synchronous-in-request heavy work and ephemeral in-memory state appear in two repos (Othor trains in the request handler past a 30s timeout; Archer's scheduler/cache/rate-limiter live in module-level memory wiped on cold start). Standardize a worker/queue + DB-or-Redis-backed scheduling triggered by real cron.
